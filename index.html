About

OntoCAAD is a proof-of-concept ontology designed as a foundation for developing collaborative AI agents that assist with architectural design tasks. It models both human and machine reasoning within the design process, grounded in empirical data from a Flemish design firm living lab and educational experiments with diffusion models.

The ontology captures four interconnected concerns: the design features embedded in artefacts, the artefacts themselves, the actors (human designers and AI agents) who create and interpret them, and the design decisions that emerge from their collaboration. It is built using the Ontology Building 101 methodology (Noy & McGuinness, 2001) and aligns human and machine reasoning through the epistemic pipeline framework (Quattrociocchi, Capraro & Perc, 2025).

Classes

Four Foundation Classes

Feature 9

Embedded design features within artefacts — the building blocks for design decisions. Style, texture, geometry, spatial relationships, feeling, lighting, details, materiality, colour scheme.

Artefact

A design artefact with typed properties: artefact type, data type, digital/iterative/external flags, embedded features, creator, and design stage.

Actor

Superclass for Person and Agent. Can create, interpret, reframe, reason, make, and implement.

DesignDecision

A decision with arguments for/against. Updates ProjectState when implemented.

Actor Hierarchy

Person

Human participant with a Role. Uses deductive, inductive, or abductive reasoning. Incorporates tacit and embodied knowledge.

Agent

Collaborative AI agent with AgentRole, DataType, and feature specialisation. Reasoning follows the epistemic pipeline.

Decision Hierarchy

DesignProject

State machine: current state, current stage, requirements list. Captures the iterative nature of design.

ProjectState

Version tracker. Updated when a DesignDecision is implemented.

DesignProblem

A requirement + intention + related features. Contains a DesignDecision.

Class Interactions

The ontology models seven class interactions — thinking processes and auxiliary functions — aligned with epistemic pipelines for both human and machine reasoning:

Thinking Processes

InteractionSignatureDescription
reframes()Actor → DesignProblemAnalyse project, identify requirement, create DesignProblem with features. Epistemic: reframing information.
reasons()Actor → DesignDecisionThree steps: (i) create artefact, (ii) interpret features, (iii) summarise arguments. Based on Schön's reflective practitioner.
makes()Actor → DesignDecisionMake a decision based on arguments for and against. Epistemic: deciding.

Auxiliary Functions

InteractionSignatureDescription
interacts()Actor ↔ ActorActors collaborate: Person–Person, Person–Agent, Agent–Agent.
implements()Actor → DesignDecisionImplement a decision, triggering ProjectState update.
creates()Actor → ArtefactPerson draws/models; Agent generates/computes.
interprets()Actor → FeatureExtract features from artefacts. Textual, visual, or spatial.

Use-Case: Facade Design

The ontology includes instance data demonstrating a complete design workflow for a residential facade in Flanders. A lead designer collaborates with a diffusion model agent to explore materiality and colour variations, ultimately selecting a vermilion red cornice — a creative suggestion the designers stated they would never have proposed themselves.

The workflow follows the four thinking processes in sequence: reframe() identifies the facade materiality problem → reason() generates four variations via the diffusion model, with the designer interpreting embedded features → make() selects the vermilion variant based on arguments → implement() updates the project state.
